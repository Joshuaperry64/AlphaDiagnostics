Blueprint: Portable, Modular Vehicle Diagnostics Tool
1. Project Vision
To create a universal, portable, and modular vehicle diagnostics tool. The system will be capable of interfacing with both modern vehicles via the OBD-II standard and older or simpler vehicles (like a golf cart) through direct sensor connections. The architecture is designed for scalability, allowing for the addition of new vehicles and sensor types in the future.
Core Components:
Hub: A Raspberry Pi 5 acting as the central processing unit, data aggregator, and web server.
Sensor Node(s): One or more ESP32 microcontrollers serving as wireless sensor platforms.
Interface: A web-based dashboard accessible from any device on the same network.
--------------------------------------------------------------------------------

2.1. Communication Protocols
OBD-II to Hub: Serial communication over USB, using an ELM327 adapter. The python-obd library will handle the protocol abstraction.
Sensor Node to Hub: MQTT over Wi-Fi. The ESP32 will publish sensor data to topics on the Mosquitto broker running on the Pi. The Pi's Python application will subscribe to these topics.
Hub to Dashboard: WebSockets for real-time data streaming and a RESTful API (FastAPI) for historical data retrieval.

3.4. Module Implementation
database.py
Defines SQLAlchemy ORM models.
Vehicle: Stores vehicle information (e.g., 'Toyota Solara 2007', 'Golf Cart').
SensorReading: Stores timestamped sensor data, linked to a Vehicle. Includes columns for sensor_name, value, and units.
Includes a function to initialize the database and create tables.
config.py
Stores all configuration variables.
DATABASE_URL = "sqlite:///./diagnostics.db"
MQTT_BROKER_ADDRESS = "localhost"
MQTT_PORT = 1883
OBD_PORT_STR = "/dev/ttyUSB0" (or auto-detected)
obd_reader.py
Contains a class OBDReader that runs in a separate thread.
Connects to the ELM327 adapter using python-obd.
Continuously polls a predefined list of OBD commands (RPM, SPEED, COOLANT_TEMP, etc.).
For each successful reading, it writes the data to the SQLite database via a thread-safe queue or direct call to a database manager function.
mqtt_client.py
Contains a class MQTTClient.
Connects to the local Mosquitto broker.
Subscribes to relevant topics (e.g., vehicles/golfcart/#).
The on_message callback parses the incoming JSON payload and stores the data in the database.
main.py
Initializes the FastAPI application.
Sets up the database connection.
Starts the OBDReader and MQTTClient in background threads on application startup.
REST API Endpoint:
GET /api/vehicles/{vehicle_id}/history: Returns historical data for a vehicle.
WebSocket Endpoint:
/ws: Streams live data from both OBD-II and MQTT sources to connected dashboard clients. It will pull the latest readings from the database or an in-memory cache.

4. Phase 2: Wireless Sensor Node (ESP32)
4.1. Hardware & Setup
Hardware: ESP32-H2 DevKitC-1, breadboard, jumper wires, sensors (e.g., voltage sensor module for battery, DS18B20 for temperature).
Software: Thonny IDE or VS Code with Pico-W-Go extension for MicroPython development.
Flashing: Flash the latest stable MicroPython firmware for the ESP32.
4.2. esp32/boot.py Implementation
Wi-Fi Connection: Connects to the specified network SSID and password.
MQTT Client Setup: Configures the MQTT client to connect to the Pi's IP address.
Sensor Reading Loop:
In an infinite loop, read values from the connected sensors (e.g., adc.read_u16() for analog sensors).
Perform necessary conversions (e.g., map ADC value to voltage).
Data Publishing:
For each reading, construct a JSON payload: {"value": 48.5, "units": "V"}.
Publish the payload to a specific MQTT topic, e.g., vehicles/golfcart/battery_voltage.
Implement a sleep in the loop to control the publishing frequency (e.g., every 5 seconds).

System Architecture & Platform Structure
- Single repository with subfolders for Pi and Windows platforms.
- Shared Python package (`alpha_common`) for protocol definitions, logging, and models; platform-specific extensions on the laptop.
- Simple requirements.txt for dependencies (no virtualenvs).

---
Communication & Data Flow
- Python used on both Pi and Windows; two interfaces communicate via REST API and/or WebSockets (FastAPI).
- Pi is primary controller for OpenPort 2.0, but device can be moved to laptop if needed.
- Pi pushes logs to laptop when reachable; laptop listens for incoming data. Use easiest transport (HTTP POST or SCP/SSH).

- Connection test, OBD diagnostics (VIN, DTCs, live sensor data), customizable features (e.g., door lock programming).
